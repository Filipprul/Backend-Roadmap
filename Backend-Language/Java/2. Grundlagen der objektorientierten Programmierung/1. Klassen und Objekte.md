# Grundidee
Eine Klasse ist kein Objekt.
Eine Klasse ist nur eine Definition, ein Bauplan.
Ein Objekt ist ein konkretes Ding, das aus diesem Bauplan erstellt wird.

Unpräzise Formulierungen wie "Klasse = Vorlage" reichen nicht.
Die korrekte technische Sicht:
- Klasse = Datentyp, selbst definiert
- Objekt = Variable, die dieser Datentyp nutzt
- Ein Objekt besitzt Zustand (Attribute) und Verhalten (Methoden)
- Die Klasse legt fest, welche Daten und welche Operationen möglich sind

# Warum brauchen wir Klassen überhaupt?
Weil Programme sonst aus unstrukturiertem Chaos bestehen würden.
Klassen lösen drei Probleme:
1. Bündelung von Daten + Verhalten
	Alles, was zusammengehört, ist in einer Einheit.
	Beispiel: Ein Spieler im Spiel
	- Daten: HP, Position, Geschwindigkeit
	- Verhalten: springen(), angreifen(), bewegen()
2. Erstellen mehrere ähnliche Objekte
	Ein Bauplan reicht → du erzeugst 200 Gegner, 5 Spieler, 1000 Kugeln.
3. Wiederverwendbarkeit
	Einmal korrekt designt → überall nutzbar

# Technische Definition (Java-Perspektive)
Klasse:
- Eine Definition im Code
- Enthält Attribute = Felder = Zustandsvariable
- Enthält Methoden = Funktionen zur Manipulation dieses Zustands

Objekt:
- Ein Stück Speicher zur Laufzeit
- Enthält konkrete Werte in den Attributen
- Führt Methoden aus, die in der Klasse definiert sind

# Was passiert beim Erstellen eines Objekts?
Beispiel:
```java
Player p = new Player();
```

Technisch passiert:
1. `new Player()` reserviert Speicherplatz für alle Attribute.
2. Der Konstruktor wird ausgeführt (auch wenn du keinen geschrieben hast).
3. Ein Objekt wird im Heap erzeugt.
4. Die Variable `p` speichert nur eine Referenz, nicht das Objekt selbst.

Wichtige Wahrheit: In Java arbeitet man immer mit Referenzen, nicht direkt mit den Objekten.

# Beispiel, das zeigt, wie es funktioniert
```java
class Player {
    int health;
    String name;

    void attack() {
        System.out.println(name + " attacks!");
    }
}
```
Erstellen:
```java
Player p1 = new Player();
p1.name = "Filipp";
p1.health = 100;

Player p2 = new Player();
p2.name = "Enemy";
p2.health = 80;

p1.attack();
p2.attack();
```
Analyse:
- p1 und p2 sind zwei verschiedene Speicherobjekte
- Beide nutzen denselben Code aus der Klasse
- Nur ihr Zustand unterscheidet sich

# Drei Grundpfeiler der OOP, die Klassen erst sinnvoll machen
1. Kapselung
	Daten geschützt + nur definierte Methoden verändern sie.
2. Vererbung
	Eine Klasse kann von einer anderen erben → Wiederverwendbarkeit + Spezialisierung.
3. Polymorphie
	Gleiche Methode - unterschiedliche Implementationen.

Ohne diese Konzepte wären Klassen unnötig Ballast.

# Typische Fehler, die man vermeiden soll
1. Eine Klasse ist ein Objekt
	→ Nein. Klasse existiert im Code. Objekt existiert im Speicher.
2. Ein Objekt ist eine Kopie in der Klasse
	→ Nein. Ein Objekt ist eine Instanz, nicht eine Kopie des Codes.
3. Mehrere Objekte teilen sich Attribute
	→ Nur statische Attribute, sonst nicht.
4. Name des Objekts = Objekt
	→ Nein. Der Name ist eine Referenzvariable.
